# Phase 4 Data Model

Extends the Phase 1/2/3 schema. Migration version bumps from v3 to
**v4**. Rollback to v3 is NOT supported — v4 is forward-only like
the earlier migrations.

## New table

### `escalation_dispatches`

One row per dispatch attempt that reached the send step (i.e.
passed both the supersession gate of FR-208 and the routing gate
of FR-202). Rows are written in the same transaction as the
`escalation_dispatched` audit event (FR-211).

| Column              | Type    | Constraints                                      | Description                                                                 |
|---------------------|---------|--------------------------------------------------|-----------------------------------------------------------------------------|
| dispatch_id         | TEXT    | PRIMARY KEY                                      | UUID v4 generated by Serbero                                                |
| dispute_id          | TEXT    | NOT NULL, FK → disputes(dispute_id)              |                                                                             |
| session_id          | TEXT    | NULL, FK → mediation_sessions(session_id)        | NULL for FR-122 dispute-scoped handoffs where no session was opened         |
| handoff_event_id    | INTEGER | NOT NULL, FK → mediation_events(id)              | The `handoff_prepared` row that triggered this dispatch                     |
| target_solver       | TEXT    | NOT NULL                                         | Pubkey (hex) of the recipient; see "Recipient encoding" below               |
| dispatched_at       | INTEGER | NOT NULL                                         | Unix seconds — when the send loop finished                                  |
| created_at          | INTEGER | NOT NULL                                         | Unix seconds — row-insert wall-clock (same as dispatched_at in practice)    |
| status              | TEXT    | NOT NULL DEFAULT 'dispatched', CHECK IN (...)    | `dispatched` \| `send_failed` — see FR-211                                  |
| fallback_broadcast  | INTEGER | NOT NULL DEFAULT 0                               | 0/1 — 1 when FR-202 rule 3 (fallback-to-all-solvers) fired                  |

**Indexes**:

- `idx_escalation_dispatches_dispute ON escalation_dispatches(dispute_id)` — operator lookup by dispute.
- `idx_escalation_dispatches_handoff ON escalation_dispatches(handoff_event_id)` — dedup probe (FR-203, SC-205).

**Recipient encoding**: `target_solver` holds a single hex pubkey
when the targeted path fired (FR-202 rule 1) and a comma-separated
list of hex pubkeys when the broadcast path fired (rule 2 or
fallback rule 3). The column type is `TEXT`, not JSON, to match the
existing `notifications.solver_pubkey` shape and avoid pulling in
JSON-functions for a simple read. Downstream operator queries can
`INSTR()` or split on the comma when they need per-recipient data.

**Status values**:

- `dispatched` — at least one recipient in `target_solver`
  successfully received the DM. `notifications` rows for the same
  dispute carry the authoritative per-recipient detail.
- `send_failed` — every recipient failed. `notifications` rows
  all carry `status = 'failed'`. SC-208 verifies the "no-JOIN"
  operator query `SELECT * FROM escalation_dispatches WHERE
  status = 'send_failed'` returns all all-failed dispatches.

**CHECK constraint**: `CHECK (status IN ('dispatched',
'send_failed'))`. Enforced at the schema level so a mis-spelled
token surfaces as a SQL error, not a silent audit-drift.

## Extension of existing table

### `mediation_events` — four new `kind` values

Appends the following variants to
`MediationEventKind` (the existing Rust enum in
`src/db/mediation_events.rs`). The string form lands as-is in the
`kind` column; the payload shape is pinned in `contracts/audit-events.md`.

| Kind                              | Meaning                                                                                                    |
|-----------------------------------|------------------------------------------------------------------------------------------------------------|
| `escalation_dispatched`           | Phase 4 delivered (or attempted to deliver) the structured handoff DM. Payload: `{dispatch_id, dispute_id, target_solver, fallback_broadcast, status}`. Dispatch-tracking row exists. |
| `escalation_superseded`           | Phase 4 skipped delivery because the dispute was already resolved when examined. Payload: `{dispute_id, reason: "dispute_already_resolved", handoff_event_id}`. No dispatch-tracking row written. |
| `escalation_dispatch_unroutable`  | FR-202 rule 4 fired: no write-permission solvers configured AND fallback off. Payload: `{dispute_id, handoff_event_id, configured_solver_count, fallback_to_all_solvers: false}`. No dispatch-tracking row; the handoff event stays unconsumed so a later config change can pick it up. |
| `escalation_dispatch_parse_failed`| FR-214: the `handoff_prepared` payload could not be used. Payload: `{dispute_id, handoff_event_id, reason: "deserialize_failed" \| "orphan_dispute_reference", detail}`. No dispatch-tracking row; the handoff event is considered consumed (manual operator action required to re-dispatch). |

Session scoping follows the established Phase 3 rule: `session_id`
is NULL when no session row exists for the dispute (FR-122
dispute-scoped handoffs) and set otherwise. The
`escalation_superseded`, `escalation_dispatch_unroutable`, and
`escalation_dispatch_parse_failed` rows copy the session_id shape
of the originating `handoff_prepared` row.

## Existing tables touched

**None are modified structurally.** Phase 4 reads from `disputes`
(for `lifecycle_state` and `assigned_solver`) and from
`mediation_events` (for `handoff_prepared` rows + dedup joins
against `escalation_dispatches`), and appends to `mediation_events`.
FR-217 forbids UPDATE / DELETE against Phase 1/2/3 tables.

## Entities (Rust domain types)

New domain types (all live under `src/escalation/` or
`src/db/escalation_dispatches.rs`):

- **`EscalationDispatch`** (mirrors the table above). Fields:
  `dispatch_id: String`, `dispute_id: String`,
  `session_id: Option<String>`, `handoff_event_id: i64`,
  `target_solver: String`, `dispatched_at: i64`, `created_at: i64`,
  `status: DispatchStatus`, `fallback_broadcast: bool`.

- **`DispatchStatus`** enum with `Dispatched` and `SendFailed`
  variants. Implements `Display` (writes `dispatched` /
  `send_failed` snake-case) and `FromStr` consistent with the
  neighboring `MediationSessionState` / `ClassificationLabel` enums.

- **`PendingHandoff`** (internal to `consumer.rs`) — the cursor row
  shape returned by the consumer scan. Fields: `handoff_event_id:
  i64`, `dispute_id: String`, `session_id: Option<String>`,
  `payload_json: String`, `occurred_at: i64`. Not persisted.

- **`DispatchOutcome`** (internal to `dispatcher.rs`) — returned by
  the send loop to the tracker. Variants: `AllSucceeded { recipients:
  Vec<String> }`, `AllFailed { attempted: Vec<String> }`,
  `PartialSuccess { succeeded: Vec<String>, failed: Vec<String> }`.
  The tracker maps the first two to `DispatchStatus::Dispatched` /
  `SendFailed` and treats `PartialSuccess` as `Dispatched` per
  FR-211.

Reuses the existing Phase 3 type:

- **`HandoffPackage`** (`src/mediation/escalation.rs`). Phase 4
  re-exports via `pub use` or imports directly; does NOT mutate
  the shape. Phase 4 treats "`session_id` key absent" and
  "`session_id: null`" as equivalent (per the 2026-04-20 data-
  model note).

## Migration sketch (for reference; not prescriptive)

```sql
-- Migration v4 body (conceptual; the real file lives in src/db/migrations.rs).
BEGIN;

CREATE TABLE IF NOT EXISTS escalation_dispatches (
    dispatch_id         TEXT PRIMARY KEY,
    dispute_id          TEXT NOT NULL,
    session_id          TEXT,
    handoff_event_id    INTEGER NOT NULL,
    target_solver       TEXT NOT NULL,
    dispatched_at       INTEGER NOT NULL,
    created_at          INTEGER NOT NULL,
    status              TEXT NOT NULL DEFAULT 'dispatched'
                        CHECK (status IN ('dispatched', 'send_failed')),
    fallback_broadcast  INTEGER NOT NULL DEFAULT 0,
    FOREIGN KEY (dispute_id) REFERENCES disputes(dispute_id),
    FOREIGN KEY (session_id) REFERENCES mediation_sessions(session_id),
    FOREIGN KEY (handoff_event_id) REFERENCES mediation_events(id)
);

CREATE INDEX IF NOT EXISTS idx_escalation_dispatches_dispute
    ON escalation_dispatches(dispute_id);
CREATE INDEX IF NOT EXISTS idx_escalation_dispatches_handoff
    ON escalation_dispatches(handoff_event_id);

UPDATE schema_version SET version = 4;

COMMIT;
```

No data backfill required. Existing `handoff_prepared` rows written
by Phase 3 are simply picked up on the first dispatcher cycle after
Phase 4 ships (subject to FR-208 supersession if the dispute has
already resolved).
